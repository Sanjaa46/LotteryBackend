Week 1 — Campaign, Prize & Code management (Admin)
Day 1–2

Campaign CRUD admin only

  • POST /api/admin/campaigns — create campaign with startDate, endDate
  • GET /api/admin/campaigns — list all, filter by status/date
  • GET /api/admin/campaigns/:id — single campaign detail
  • PATCH /api/admin/campaigns/:id/status — DRAFT → ACTIVE → PAUSED → ENDED
  • LEARN Admin role middleware — check req.user.role === 'admin'
Day 3

Prize management

  • POST /api/admin/campaigns/:id/prizes — add prize with probability %
  • GET /api/admin/campaigns/:id/prizes — list prizes + remainingQuantity
  • PATCH /api/admin/prizes/:id — edit name, type, quantity, probability
  • LEARN Store probability as integer (e.g. 30 = 30%). Sum must ≤ 100
Day 4–5

Lottery code generation (bulk)

  • POST /api/admin/campaigns/:id/codes/generate — body: { count: 1000 }
  • LEARN Prisma transaction: createMany CodeBatch + LotteryCode in one tx
  • LOGIC Assign prizeId by probability — weighted random, respect remainingQuantity
  • GET /api/admin/batches/:id/export — stream CSV download (csv-stringify)
  • GET /api/admin/campaigns/:id/batches — list all code batches
Week 2 — Code submission, Lottery logic & Prize claim (User + Admin)
Day 6–7

Code submission endpoint

  • POST /api/lottery/submit — body: { code: "ABC123" }
  • VALIDATE Code exists → campaign is ACTIVE → isUsed === false
  • LEARN Prisma transaction: mark isUsed=true + usedAt, create Submission atomically
  • REDIS Rate limit: max 10 submits/hour per IP · max 5 per user per day
  • LOGIC Check prizeId on code → if set: WIN, else: LOSE
Day 8

Win/Lose result & PrizeClaim creation

  • WIN Decrement Prize.remainingQuantity (use Prisma atomic decrement)
  • WIN Create PrizeClaim { status: PENDING } linked to Submission
  • LOSE No PrizeClaim created — return { result: "LOSE" } response
  • LEARN Save AuditLog for every submission — userId, action, IP, timestamp
  • EMAIL Send win notification email via existing mailer.ts
Day 9–10

Prize claim management & user history

  • GET /api/user/submissions — my code submission history + WIN/LOSE result
  • GET /api/user/claims — my prize claims + status (PENDING/APPROVED/DELIVERED)
  • GET /api/admin/claims — all claims, filter by status, date, campaign
  • PATCH /api/admin/claims/:id/status — approve / reject / mark delivered
  • EMAIL Notify user on claim status change (approved or delivered)
Week 3 — Dashboard, Reports & Security hardening
Day 11–12

Admin dashboard statistics

  • GET /api/admin/dashboard — total codes, used codes, winner count, claim stats
  • GET /api/admin/dashboard?campaignId=1 — per-campaign breakdown
  • REDIS Cache dashboard result with TTL 60s — avoid heavy DB aggregations on every request
  • LEARN Prisma groupBy + _count + _sum for aggregation queries
Day 13

Report export (CSV)

  • GET /api/admin/reports/submissions?campaignId=1&from=...&to=... → CSV
  • GET /api/admin/reports/winners?campaignId=1 → CSV with prize details
  • LEARN Use csv-stringify or exceljs — stream large result sets, don't load all into memory
  • LEARN Set res.setHeader('Content-Disposition', 'attachment; filename=report.csv')
Day 14–15

Security hardening & final cleanup

  • PKG helmet — secure HTTP headers on all routes
  • REDIS express-rate-limit + RedisStore — global rate limit on all public routes
  • GUARD CAPTCHA token validation on POST /lottery/submit
  • GUARD Admin middleware on every /api/admin/* route — 403 if not admin role
  • AUDIT Verify all write actions have AuditLog entries with userId + ipAddress
